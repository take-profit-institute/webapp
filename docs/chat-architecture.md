# 종목 채팅 아키텍처 (실시간 채팅)

> 종목별 실시간 채팅의 설계 결정 기록. **메시지를 영속 저장하지 않는** 전제 위에서,
> WebSocket + Redis Pub/Sub으로 실시간 서빙(라우팅)에만 집중하는 구조다.
> 시세 스트림은 [`realtime-architecture.md`](./realtime-architecture.md) 참고.

## 개요 / 전제

| 전제 | 영향 |
|------|------|
| **채팅 내용을 저장하지 않음** | 데이터 정합성·순서 보장·DB 동기화 불필요 → 난이도 대폭 하락. 실시간 라우팅에만 집중 |
| **종목당 방을 500명 단위로 쪼갬** | 방 메타데이터(인원수)를 인스턴스 간 공유해야 함 → Redis 카운터 사용 |
| **WS 서버는 Stateless** | 커넥션·방 상태는 전부 Redis에 위임. 인스턴스는 언제 죽어도 됨 |

핵심 기술 선택: **WebSocket(전달) + Redis Pub/Sub(팬아웃) + Redis 카운터(방 배정)**.

## 전체 아키텍처 및 흐름

```
[클라이언트]
   │ 1. 방 배정 요청 (REST)         /chat/rooms?symbol=005930
   ▼
[게이트웨이(8000)] ── JWT 검증 → X-Account-Id 주입
   │
   ├─(REST)→ [방 배정 서비스]  ── Redis 카운터 조회/INCR → 종목코드_방번호 반환
   │
   │ 2. 반환받은 방 ID로 WS 연결    wss://.../chat/ws?room=005930_1&token=...
   ▼
[게이트웨이(8000)] ── 핸드셰이크에서 JWT 검증 → X-Account-Id 주입
   │
   ▼
[chat-gateway (WS 커넥션 서버, N대)]
   │   - 연결 시 Redis INCR 005930_1_count
   │   - Redis SUBSCRIBE chat:005930_1
   │
   ▼ 3. 메시지 송수신
[Redis Pub/Sub]  chat:005930_1 채널
   ▲ PUBLISH                  │ 브로드캐스트
   └── 보낸 WS 서버 ──────────┴──→ 같은 채널 구독 중인 모든 WS 서버 → 각자 붙은 유저에게 전달
```

1. 클라이언트가 종목 방 진입을 요청한다.
2. 방 배정 서비스가 현재 활성 인원 기반으로 `종목코드_방번호`를 지정한다.
3. 클라이언트는 그 방 ID로 WS 서버에 커넥션을 맺는다.
4. WS 서버는 `chat:종목코드_방번호` 채널을 Subscribe하고, 받은 메시지를 같은 채널로 Publish/브로드캐스트한다.

## 방 쪼개기 (500명 단위) 전략

이 시스템의 유일한 기술적 챌린지는 "인원수를 실시간 공유해서 방을 배정"하는 것이다. MSA라 여러 WS 서버가 인원수를 공유해야 하므로 **Redis**에 위임한다.

### Redis presence (ZSET + per-멤버 TTL, 자가치유)

> 초기엔 `종목코드_방번호_count` 단일 키 `INCR`/`DECR`를 썼으나, 비정상 종료(`DECR` 누락)와
> 재연결(중복 `INCR`)로 카운터가 영구히 드리프트하는 문제가 있어 presence ZSET으로 교체했다.

- 방마다 ZSET `종목코드_방번호_presence` — member = 커넥션 id(`{accountId}:{uuid}`), score = 마지막 heartbeat epoch millis
- 입장: `ZADD`(score=now) → 퇴장(정상 종료): `ZREM`
- **heartbeat**: 커넥션이 살아있는 동안 주기적으로 `ZADD`로 score 갱신 (`CHAT_HEARTBEAT`, 기본 30s)
- **활성 인원** = score ≥ `now - presenceTtl`(`CHAT_PRESENCE_TTL`, 기본 90s)인 멤버 수. 조회 전
  `ZREMRANGEBYSCORE`로 만료분을 정리한 뒤 `ZCARD`. → heartbeat가 끊긴(크래시/네트워크 단절) 멤버는
  TTL 경과 후 자동 제외되어 `DECR` 누락이 있어도 인원이 새지 않는다(자가치유).
- 입장/퇴장마다 정리 후 인원수를 채널로 broadcast → 방의 모든 클라가 실시간 반영(크래시 드리프트는
  다음 입/퇴장 이벤트에서 재동기화된다).

### 방 배정 로직

```
유저가 '삼성전자(005930)' 입장 요청
  1. Redis에서 005930의 활성 방 카운트 조회
  2. 005930_1 이 450명 → 거기 배정
  3. 모든 방이 500 꽉 참 / 방 없음 → 005930_2 발급, 카운터 1로 세팅
```

### 동시성 (Race Condition)

천 명이 동시에 같은 종목으로 들이닥치면 순간적으로 한 방에 510명이 들어갈 수 있다.

| 정책 | 방식 | 비고 |
|------|------|------|
| **엄격한 500명 보장** | Redis 분산 락(Redisson 등)으로 배정 로직 잠금 | 정확하지만 느림 |
| **대략 500명 안팎 허용** (채택) | 락 없이 `INCR` 후 리턴값이 500 초과면 다음 방으로 토스 (낙관적) | 가볍고 충분. 주식 채팅 특성상 ±α 허용 |

> 결정: 주식 채팅방은 "대충 500명 안팎"이면 되므로 **낙관적 방식**으로 간다. 엄격 보장이 필요해지면 락 도입.

## 스케일아웃: Redis Pub/Sub 팬아웃

WS 서버를 3대로 늘렸을 때:

- 유저 A는 1번 서버, 유저 B는 2번 서버에 붙어 있고 둘 다 `005930_1` 방.
- A가 메시지 전송 → 1번 서버가 `chat:005930_1` 채널로 **PUBLISH**.
- Redis가 브로드캐스트 → `005930_1` 구독 중인 **모든 서버(1,2,3)** 가 수신 → 각자 붙은 유저에게 전달.

→ 서버가 몇 대로 늘어나도 Pub/Sub이 알아서 공유한다.

## 게이트웨이를 끼고 있을 때의 설계 결정

현재 구조: **게이트웨이(8000) → BFF(4000) → gRPC 마이크로서비스**. 게이트웨이가 REST에서 JWT 검증 후 `X-Account-Id`를 주입하는 흐름이 이미 존재한다(시세 WS 및 `/users/me` 등과 동일).

### 결정 1 — WS 커넥션 서버 위치: 전용 서비스 분리

| 선택지 | 장점 | 단점 |
|--------|------|------|
| **A. BFF에 채팅 WS 라우트 추가** | 기존 `@fastify/websocket` 재사용, 배포 1개. 빠른 PoC | BFF가 stateful(롱커넥션)해짐. 채팅 부하로 REST까지 출렁임 |
| **B. 전용 `chat-gateway` 서비스 (채택)** | 커넥션 부하를 BFF와 격리, 독립 스케일아웃. "WS는 Stateless, 상태는 Redis" 원칙과 일치 | 서비스 1개 추가 |

> 결정: **B 채택.** 단, 급하게 검증만 할 거면 A로 BFF `ws.routes.ts`에 임시로 붙였다가 떼어내는 경로도 허용.

> ⚠️ 참고: 기존 시세 WS(`bff/src/services/market-stream.service.ts`)는 클라이언트를 **프로세스 메모리 `Map`** 에 들고 직접 `send`하는 **스케일아웃 불가** 구조다(현재 mock이라 무방). 채팅은 이를 확장하는 게 아니라 Redis Pub/Sub 팬아웃을 새로 얹는다. 추후 시세 WS도 같은 패턴으로 교체하면 스케일아웃 가능.

### 결정 2 — sticky session 불필요

게이트웨이/LB가 유저를 어느 WS 인스턴스에 붙이든 무관하다. 모든 인스턴스가 같은 채널을 Subscribe하므로 어디 붙어도 같은 메시지를 받는다. → **세션 어피니티 설정 안 함.** (Pub/Sub을 고른 진짜 보상)

단, 게이트웨이/인그레스가 **WebSocket Upgrade를 프록시**할 수 있어야 하고 **idle timeout을 충분히 길게** 잡아야 한다(롱커넥션 보호). Spring Cloud Gateway는 ws/wss 라우팅 기본 지원.

### 결정 3 — 인증은 핸드셰이크 시점 1회

WS도 **최초 Upgrade 요청(GET)** 에 대해 게이트웨이가 JWT 검증 → `X-Account-Id` 주입 → WS 서버는 `connection` 핸들러에서 그 헤더로 유저 확정.

> ⚠️ 함정: 브라우저 `WebSocket` API는 커스텀 헤더(`Authorization`)를 못 붙인다. 토큰 전달 방법:
> - `wss://.../chat/ws?token=<accessToken>` 쿼리스트링, 또는
> - `Sec-WebSocket-Protocol` 서브프로토콜에 토큰 적재
>
> 쿼리 토큰은 로그에 남을 수 있으니 **짧은 수명 토큰** 권장.

### 결정 4 — 트래픽 2갈래 라우팅

| 경로 | 프로토콜 | 대상 |
|------|----------|------|
| `/chat/rooms/**` | REST | 방 배정 서비스 (Redis 카운터 조회/INCR) |
| `/chat/ws` | WebSocket Upgrade | chat-gateway (WS 서버) |

게이트웨이에 이 2개 라우트를 정리한다(`/chat/ws`는 idle timeout 길게).

## Edge Case: 커넥션 끊김 처리

사용자가 터널 진입으로 끊기거나 브라우저를 그냥 닫으면 정상 `onClose`가 안 올 수 있다.

- 정상 종료: WS 종료 시 presence ZSET에서 `ZREM`(즉시 반영).
- 비정상 종료: `ZREM`이 누락돼도 heartbeat가 끊긴 멤버의 score가 갱신되지 않아 `presenceTtl` 경과 후
  집계에서 자동 제외된다(자가치유). → "유저 없는데 방이 꽉 찼다" 오인/새 방 남발이 구조적으로 방지됨.
- 즉 정합성은 heartbeat + per-멤버 TTL로 보장하며, 별도 정합성 배치가 필요 없다.

## Redis Pub/Sub의 한계 인지

- Pub/Sub은 메시지를 큐에 저장하지 않는 **Fire-and-Forget** → 매우 빠름. 저장 안 하는 본 설계에 최적.
- Redis는 싱글 스레드라 **채널 수 폭증**(수천 종목 × 쪼개진 방)이면 CPU가 튈 수 있음 → 모니터링 필요.
- 끊긴 동안의 메시지는 유실(저장 안 하므로 의도된 동작).

## Redis 키 / 채널 설계

| 종류 | 키/채널 | 용도 |
|------|---------|------|
| Pub/Sub 채널 | `chat:{종목코드}_{방번호}` | 방 단위 메시지 브로드캐스트 |
| presence (ZSET) | `{종목코드}_{방번호}_presence` | member=커넥션, score=last-seen. 만료 정리 후 `ZCARD`로 인원 추적, 배정 기준 |
| 방 시퀀스 | `chat:rooms:{종목코드}` | 종목별 발급된 최대 방 번호(`INCR`) |

## 난이도 / 요약

- **난이도:** 중하 (메시지 영속 저장 안 하므로 정합성·순서 보장 지옥을 회피)
- **핵심 과제:** Redis 카운터로 유저를 500명 단위로 라우팅하는 서빙 레이어
- **인프라 원칙:** WS 서버 Stateless, 룸 메타데이터·동기화는 전부 Redis 위임
- **게이트웨이 관점 추가분:** 사실상 "WS Upgrade 라우트 + 핸드셰이크 인증" 한 덩어리뿐
